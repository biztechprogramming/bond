08 — Sandbox System

Named sandbox profiles, assignable to agents, skills, and channels. Right tools, zero excess.

Agent Zero: Docker-first executionOpenClaw: per-session sandbox configNew: multi-profile sandbox assignment

The Problem

A "one sandbox fits all" model forces a choice: either the sandbox has everything (insecure, bloated) or it has the minimum (useless for most tasks). Different work needs different environments:

  • A coding agent needs Node, Python, git, compilers — but not email access or browser
  • A research agent needs browser and web search — but not shell access
  • A data analysis skill needs pandas, matplotlib, jupyter — but not network access
  • Messages from Discord groups should run in a locked-down sandbox with no file access outside workspace
  • Your main session might not need a sandbox at all

Solution: Named Sandbox Profiles

Bond uses named sandbox profiles — each is a defined environment with specific tools, network access, and filesystem mounts. Profiles are then assigned to agents, skills, channels, or sessions.

  Sandbox Profiles (defined once)
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │   coding     │ │   research   │ │   data       │ │   minimal    │
  │              │ │              │ │              │ │              │
  │ node, python │ │ browser      │ │ python       │ │ (empty)      │
  │ git, gcc     │ │ web_search   │ │ pandas, plt  │ │              │
  │ filesystem   │ │ filesystem   │ │ jupyter      │ │ read-only fs │
  │ network: lan │ │ network: web │ │ network: off │ │ network: off │
  └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
         │                │                │                │
         ▼                ▼                ▼                ▼
  Assignments (who uses what)
  ┌──────────────────────────────────────────────────────────────────┐
  │  agent "coder"        → sandbox: coding                         │
  │  agent "researcher"   → sandbox: research                       │
  │  skill "data-analysis"→ sandbox: data                           │
  │  channel discord      → sandbox: minimal                        │
  │  channel telegram     → sandbox: minimal                        │
  │  main session         → sandbox: none (host)                    │
  └──────────────────────────────────────────────────────────────────┘

Configuration

The user experience is the priority. Configuration should be obvious and short.

Step 1: Define profiles

Profiles live in bond.json under sandboxes. Each profile says what's inside and what's allowed.

{
  "sandboxes": {
    // Profile name → environment definition

    "coding": {
      "image": "bond-coding",        // Docker image (or Dockerfile path)
      "packages": ["nodejs", "python3", "git", "gcc", "make"],
      "network": "lan",              // "off" | "lan" | "web" | "full"
      "filesystem": "workspace",     // "none" | "readonly" | "workspace" | "full"
      "memory": "2g",
      "timeout": 300
    },

    "research": {
      "packages": ["chromium"],
      "tools": ["browser", "web_search", "web_fetch"],
      "network": "web",
      "filesystem": "readonly"
    },

    "data": {
      "image": "bond-data",
      "packages": ["python3", "python3-pip"],
      "pip": ["pandas", "matplotlib", "numpy", "jupyter"],
      "network": "off",
      "filesystem": "workspace"
    },

    "minimal": {
      "packages": [],
      "network": "off",
      "filesystem": "none",
      "tools": ["respond", "search_memory"]
    }
  }
}
💡 Easy mode: just name the packages

You don't need to write Dockerfiles. List the packages you want and Bond builds the image automatically. For power users, point image at a custom Docker image or a Dockerfile in your workspace.

Step 2: Assign profiles

Profiles are assigned at multiple levels. More specific assignments override less specific ones.

{
  // Default sandbox for all non-main sessions
  "sandbox": "minimal",

  "agents": {
    "coder": {
      "profile": "coding-agent",
      "sandbox": "coding"           // this agent always uses "coding"
    },
    "researcher": {
      "profile": "research-agent",
      "sandbox": "research"
    }
  },

  "channels": {
    "discord": {
      "sandbox": "minimal"          // all Discord sessions use "minimal"
    },
    "telegram": {
      "sandbox": "minimal"
    },
    "webchat": {
      "sandbox": "none"             // webchat = trusted, no sandbox
    }
  },

  "skills": {
    "data-analysis": {
      "sandbox": "data"             // this skill runs in "data" environment
    },
    "web-scraper": {
      "sandbox": "research"
    }
  }
}

Step 3: That's it

Bond resolves the sandbox at runtime based on the context. No manual Docker commands needed.

Resolution Order

When code needs to execute, Bond determines the sandbox by checking assignments in priority order:

  Code execution requested
      │
      ├─ 1. Is a specific skill active? → use skill's sandbox
      │
      ├─ 2. Is the agent assigned a sandbox? → use agent's sandbox
      │
      ├─ 3. Is the channel assigned a sandbox? → use channel's sandbox
      │
      ├─ 4. Is there a session-level override? → use that
      │
      ├─ 5. Is this the main session? → "none" (host) by default
      │
      └─ 6. Fallback → global default sandbox

The most specific match wins. A skill's sandbox requirement always takes priority — if a skill says it needs the "data" sandbox, it gets it regardless of what channel triggered it.

Skill Sandbox Declarations

Skills can declare their sandbox requirements directly in SKILL.md. Bond reads this and auto-creates a matching profile if one doesn't exist.

# SKILL.md — data-analysis

## Metadata
name: data-analysis
version: 1.0.0

## Sandbox
requires:
  packages: [python3, python3-pip]
  pip: [pandas, matplotlib, numpy, seaborn]
  network: "off"
  filesystem: "workspace"
  memory: "1g"

When a skill declares sandbox requirements:

  1. Bond checks if a compatible profile already exists
  2. If not, it auto-creates one (named skill-{skill-name})
  3. The image is built on first use and cached
  4. The user can override by explicitly assigning a different sandbox in bond.json
🔒 Skill sandboxes are security boundaries

When you install a skill from a registry, its code runs in the sandbox it declares — not on your host. A malicious skill with network: "off" and filesystem: "workspace" can't phone home or read your files. This makes skill installation much safer.

Agent Profiles + Sandboxes

Agent profiles (from Agent Zero) naturally pair with sandboxes. Each agent profile defines persona, model, and now also execution environment.

{
  "agents": {
    "coder": {
      "profile": "You are a senior software engineer...",
      "model": "anthropic/claude-sonnet-4-20250514",
      "sandbox": "coding",
      "tools": ["code_execute", "file_read", "file_write", "search_memory", "respond"]
    },

    "researcher": {
      "profile": "You are a thorough research assistant...",
      "model": "anthropic/claude-sonnet-4-20250514",
      "sandbox": "research",
      "tools": ["browser", "web_search", "web_fetch", "search_memory", "respond"]
    },

    "analyst": {
      "profile": "You are a data analyst...",
      "model": "openai/gpt-4o",
      "sandbox": "data",
      "tools": ["code_execute", "file_read", "file_write", "respond"]
    }
  }
}

When Agent 0 spawns a subordinate with call_subordinate(profile: "coder"), the coder agent automatically runs in the "coding" sandbox. The parent agent doesn't need to know or care about sandbox details.

Channel Sandboxes

Channels carry different trust levels. A message from your private webchat is very different from a message in a public Discord server.

{
  "channels": {
    // Trusted channels: no sandbox (or light sandbox)
    "webchat": { "sandbox": "none" },
    "signal":  { "sandbox": "none" },

    // Semi-trusted: workspace-scoped sandbox
    "telegram": { "sandbox": "minimal" },
    "whatsapp": { "sandbox": "minimal" },

    // Untrusted: locked-down sandbox
    "discord": {
      "sandbox": "minimal",
      "groups": {
        "*": { "sandbox": "minimal" }   // all groups
      },
      "dm": {
        "sandbox": "coding"              // DMs get more access
      }
    }
  }
}

You can even set different sandboxes for DMs vs groups on the same channel.

Sandbox Lifecycle

  First use of profile "coding"
      │
      ├─ Profile has custom image? → pull/build it
      │
      ├─ Profile has packages only? → auto-generate Dockerfile
      │     FROM ubuntu:24.04
      │     RUN apt-get install -y nodejs python3 git gcc make
      │     → build and tag as bond-sandbox-coding
      │
      ├─ Cache the built image
      │
      └─ On code execution:
           ├─ persistent=true  → start container, keep alive between commands
           └─ persistent=false → fresh container per command, destroy after

  Image rebuild triggers:
      • Profile packages/pip/npm changed in bond.json
      • User runs: bond sandbox rebuild coding
      • Skill sandbox requirements changed

CLI Commands

# List all sandbox profiles and their assignments
bond sandbox list

# Show details of a specific profile
bond sandbox inspect coding

# Rebuild a sandbox image (after changing packages)
bond sandbox rebuild coding

# Rebuild all sandbox images
bond sandbox rebuild --all

# Test a sandbox (opens interactive shell)
bond sandbox shell coding

# Show which sandbox would be used for a given context
bond sandbox resolve --channel discord --agent coder --skill data-analysis
# → skill "data-analysis" declares sandbox "data" (priority: skill)

Web UI

The Settings page includes a Sandboxes section:

  • Profile cards — each profile shown with its packages, network/fs settings, and a list of what's assigned to it
  • Create profile — form with package picker, network/fs dropdowns, resource sliders
  • Assignment view — see at a glance: which agent/skill/channel uses which sandbox
  • Drag-and-drop — assign a sandbox to an agent/channel by dragging the profile onto it
  • Build status — shows whether the image is built, building, or needs rebuild

Design Decisions

DecisionRationale
Skill sandbox takes prioritySkills come from registries — their security boundary must be enforced regardless of who triggered them
Packages, not Dockerfiles90% of users shouldn't need to write Docker anything. List packages, Bond handles the rest.
Prebuilt defaultsWorks out of the box. No configuration required for basic use. Power users customize.
"none" for main sessionThe owner should have full host access by default. Sandboxing yourself creates friction without security benefit.
Per-channel granularityA DM from a known contact and a message in a public Discord server have very different trust levels.
Persistent container optionSome workflows (like running a dev server) need state between commands. Others need clean isolation each time.
Auto-build from skill declarationsInstalling a skill should Just Work™. The sandbox is part of the skill's contract.
