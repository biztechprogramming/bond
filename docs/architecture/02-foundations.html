02 — Shared Foundations

Subsystems that multiple features depend on. Each is standalone with a clear interface.

F1: Knowledge Store

Used by: Memory, RAG Context, Proactive Intelligence, Task Engine, Inbox Triage
AgentClaw: design EmailPipeline: SQLite + sqlite-vec + FTS implementation Agent Zero: persistent memory storage

A unified local storage layer combining:

  • SQLite — relational data (tasks, entities, relationships, audit log)
  • sqlite-vec — vector similarity search over embeddings
  • FTS5 — full-text search fallback and keyword boosting
  • Blob store — file-system backed, content-addressed (screenshots, attachments)

Single write path. All writes go through mediator commands so pipeline behaviors (validation, auditing) apply uniformly. From EmailPipeline we take the proven repository pattern that hides database-specific vector operations. From Agent Zero we take the persistent memory approach where solutions and facts survive across sessions.

F2: Entity Graph

Used by: Memory, Proactive Intelligence, Task Engine, Inbox Triage
AgentClaw: entity-relationship model EmailPipeline: email → entity extraction

An entity-relationship layer on top of the Knowledge Store. Auto-extracted from conversations, files, emails, and calendar events.

Entity types: Person, Project, Task, Decision, Meeting, Document, Event.

Relationships are typed and timestamped: Person —works_on→ Project, Task —blocked_by→ Task. Extraction runs as a notification handler on ContentIndexed events — asynchronous, never blocking the main pipeline.

F3: Behavior Model

Used by: Predictive Actions, Persona Engine, Context Management
AgentClaw: behavior model design Agent Zero: learning from user interactions

A learned model of the user's patterns and preferences, built from observation over time. Stores:

  • Routine sequences — ordered action patterns with time correlation
  • Communication style — per-channel tone/formality profiles
  • Decision patterns — how the user responds to categories of decisions

This is structured data injected into prompts as context — not a fine-tuned model. The LLM interprets the patterns; the Behavior Model just captures them.

F4: Capability Gate

Used by: All features (via pipeline behavior)
AgentClaw: capability-based security OpenClaw: exec guards, DM policy, tool allowlists

The authorization layer. Every mediator request passes through it.

  • Capabilities — granular permissions: exec.shell, file.read, email.send, message.send.external
  • Contexts — scope capabilities: channel, persona, time-of-day, sensitivity
  • Policies — map contexts to capability sets
  • Approval flows — when a command requires a capability the current context doesn't have, prompt for approval instead of hard-denying

All decisions logged to the audit trail. Combines OpenClaw's proven DM pairing and exec sandboxing with AgentClaw's more granular capability model.

F5: Skills Engine

Used by: Agent Runtime, all extensible features
Agent Zero: SKILL.md standard, dynamic loading OpenClaw: managed skill lifecycle, ClawHub

Skills are portable folders with a SKILL.md file defining instructions, scripts, and assets. The Skills Engine:

  • Discovers and lists available skills
  • Loads skill instructions into agent context on demand
  • Manages skill installation from registries
  • Provides the agent with skills:list and skills:load operations

Compatible with the open SKILL.md standard (works with Claude Code, Codex, Cursor, and more). From OpenClaw we take the managed lifecycle (install, update, remove). From Agent Zero we take the dynamic loading approach where skills are loaded only when relevant.

F6: Embeddings Engine

Used by: Knowledge Store, Inbox Triage, Classification
EmailPipeline: local sentence-transformers, provider abstraction

Pluggable embedding generation with a local-first default:

  • Default: sentence-transformers/all-MiniLM-L6-v2 (384 dims, runs locally, free)
  • Optional: OpenAI text-embedding-3-small if API key provided
  • Batch processing (32 items/batch) for efficiency
  • Tracks which model generated each embedding for future re-embedding

Provider abstraction from EmailPipeline — swap models without touching business logic.
