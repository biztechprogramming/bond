07 — Security Model

Capability-based access, progressive trust, sandboxing, and credential management.

OpenClaw: exec guards, DM pairing, sandboxAgentClaw: capability gate, audit trailEmailPipeline: credential encryption, JWT auth

Philosophy

Security is a deliberate tradeoff: strong defaults without killing capability. Bond is powerful for real work while making risky paths explicit and operator-controlled. This comes directly from OpenClaw's approach — proven across thousands of deployments.

Defense Layers

Layer 1: Capability Gate

Every mediator request passes through the Capability Gate behavior. Granular permissions checked against the current context.

All decisions logged to the immutable audit trail.

Layer 2: Progressive Trust

From AgentClaw's design: capabilities expand as the user grants more access. Nothing assumes full permissions.

  • New user / unknown DM — pairing code required (from OpenClaw). Bot doesn't process until approved.
  • Allowlisted user — basic conversation, memory, search. No exec, no email.
  • Main session (owner) — full access: exec, browser, email, all tools.
  • Elevated mode — opt-in per session for sudo-level operations.

Layer 3: Execution Sandbox

Bond uses named sandbox profiles — tailored Docker environments assigned per-agent, per-skill, and per-channel. See 08 — Sandbox System for full details.

  • Main session — no sandbox (host access) by default
  • Agent profiles — each agent type gets a sandbox with exactly the tools it needs (coder → coding sandbox, researcher → research sandbox)
  • Skills — declare sandbox requirements in SKILL.md; Bond auto-provisions the environment. Installed skills always run in their declared sandbox.
  • Channels — map to sandboxes by trust level. Discord groups → locked-down. Your webchat → open.
  • Priority — skill sandbox > agent sandbox > channel sandbox > session default

Users configure by listing packages — no Dockerfiles needed. Prebuilt profiles (coding, research, data, minimal) work out of the box.

Layer 4: Credential Security

Bond uses a dedicated encrypted vault for all secrets — they never appear in config files. See 11 — Configuration for the full secret management design.

  • Encrypted vault — ~/.bond/data/credentials.enc, Fernet-encrypted. API keys, bot tokens, OAuth tokens all live here.
  • OS keychain for vault key — the master encryption key lives in macOS Keychain / Linux Secret Service / Windows Credential Manager. Auto-generated, no manual key management.
  • Validated on entry — every credential is tested before storing. Bad keys are caught immediately, not at 3 AM when a cron job fails.
  • Config references names, not secrets — bond.json says "provider": "anthropic", not the actual key. Safe to commit to git.
  • Masked everywhere — Settings UI, logs, API responses never expose raw secrets.
  • Env var override — for Docker/CI. Env vars take priority over vault when present.

Layer 5: Network Security

From OpenClaw:

  • Loopback binding — gateway binds to 127.0.0.1 by default
  • Tailscale integration — Serve (tailnet-only) or Funnel (public) with automatic HTTPS
  • Password auth required — any public exposure requires password authentication
  • DM pairing — unknown senders get a pairing code, bot doesn't engage until approved

Audit Trail

Every capability decision, every command execution, every external action is recorded in an append-only audit log. This is critical for:

  • Debugging what happened and when
  • Understanding what the agent did with your access
  • Post-incident review
  • Building trust over time (the agent earns trust through transparent history)
  audit_log
  ├── timestamp: 2026-02-19T08:30:00Z
  ├── command: ExecuteCode
  ├── actor: agent-main
  ├── capability: exec.shell
  ├── context: { channel: "webchat", session: "main" }
  ├── args: { code: "ls -la ~/projects" }
  └── result: success

Threat Model

Credential exfiltrationEncrypted at rest, never in agent context, masked in UI
Agent sends unauthorized messagesmessage.send capability requires owner approval
Data leak to group chatPrivate data (MEMORY.md, emails) only loaded in main session
Unauthorized remote accessLoopback binding + Tailscale + password auth
Sub-agent privilege escalationSub-agents inherit ≤ parent capabilities, never more

Safe Defaults

Bond ships with restrictive defaults that users explicitly relax:

  • DM policy: pairing (unknown users get pairing code)
  • Group activation: mention (bot only responds when mentioned)
  • External actions (email, message): require confirmation
  • Network: loopback only
  • Non-main sessions: sandboxed

The goal: you should never be surprised by what Bond does. Powerful when you want it, restrained by default.
