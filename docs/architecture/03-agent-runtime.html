Agent Loop

Bond's agent loop combines OpenClaw's session-based RPC model with Agent Zero's organic, self-directing approach. Each agent turn:

  1. Receive message (from user, superior agent, or system event)
  2. Retrieve relevant context via SearchMemory query (RAG)
  3. Load applicable skills via Skills Engine
  4. Generate response with tool calls
  5. Execute tool calls through the mediator pipeline
  6. Emit notifications for memory indexing, proactive evaluation
  7. Respond to user/superior, or delegate to subordinate

  │           Agent Turn                  │

  │  5. Emit notifications                │

Multi-Agent Hierarchy

From Agent Zero: every agent has a superior (the first agent's superior is the human). Any agent can spawn subordinates for subtasks. This keeps context focused and enables specialization.

  Human User
      │
      ▼
  Agent 0 (main)
      │
      ├── Agent 1 (researcher) ──── "Find pricing data for competitors"
      │       │
      │       └── Agent 2 (coder) ── "Scrape and format the data"
      │
      └── Agent 3 (writer) ──────── "Draft the report using the data"

Key principles:

  • Delegate subtasks, not the whole task — subordinates get specific, scoped work
  • Reset vs. continue — spawn a fresh subordinate or continue talking to an existing one
  • Agent profiles — subordinates can be specialized (coder, researcher, writer) with different system prompts and model configs
  • Clean reporting — subordinates report results back to their superior; the superior synthesizes

Code Execution Sandbox

From Agent Zero: the agent uses the computer as a tool. It can write and run code to accomplish tasks. Bond uses named sandbox profiles — tailored execution environments assigned to agents, skills, and channels.

  • Each agent profile gets its own sandbox — a coder agent gets Node + Python + git; a researcher gets a browser; a data analyst gets pandas + matplotlib
  • Skills declare their sandbox requirements — install a skill and its environment is auto-provisioned
  • Channels map to sandboxes by trust level — Discord groups get a locked-down sandbox; your private webchat runs on host
  • Main session — no sandbox by default (full host access for the owner)

Users configure sandboxes by listing packages — no Dockerfiles needed. Bond builds images automatically. See 08 — Sandbox System for full details.

  Agent "coder"   → sandbox: coding   (node, python, git, network: lan)
  Agent "research" → sandbox: research (browser, network: web)
  Skill "data"    → sandbox: data     (pandas, matplotlib, network: off)
  Discord groups  → sandbox: minimal  (empty, network: off)
  Main session    → sandbox: none     (host, full access)

Model Provider Support

The Python backend never calls LLMs directly. All LLM requests are routed through the TypeScript gateway, which owns authentication, streaming, and provider management via pi-agent-core and pi-ai (from OpenClaw). This is a deliberate architectural decision to preserve OAuth subscription support — the killer feature that lets users authenticate with their existing Claude Pro/Max or ChatGPT Plus/Pro subscriptions instead of needing API keys.

Auth Profile System

From OpenClaw: the AuthProfileStore manages multiple credential profiles per provider with:

  • OAuth profiles — subscription-based auth with automatic token refresh
  • API key profiles — standard API key authentication
  • Rotation — distribute load across profiles
  • Cooldowns — back off from rate-limited profiles
  • Failover — OAuth primary → API key fallback
  • Usage tracking — per-profile usage metrics

Per-session model overrides, thinking level control, and automatic failover between providers.

Session Model

From OpenClaw: sessions are the unit of agent context. Each session has its own history, model config, and capability scope.

  • Main session — direct chat with the user, full access
  • Group sessions — isolated per group/channel, reduced capabilities
  • Spawned sessions — sub-agent runs, isolated context, report back when done
  • Cron sessions — isolated runs triggered by schedule
