01 — Mediator Pipeline

The central nervous system — every operation flows through here.

from AgentClaw: MediatR designfrom EmailPipeline: Python mediator + behaviors

Why a Mediator

Direct service-to-service calls create invisible coupling. Adding a feature means touching existing code paths. The mediator gives us a single dispatch pattern for commands, queries, and notifications with a pipeline that handles cross-cutting concerns uniformly.

Bond combines the TypeScript MediatR approach from AgentClaw with the proven Python pipeline behaviors from EmailPipeline. The result: typed, ordered, testable middleware for every operation — regardless of language choice.

Core Primitives

Commands (write operations)

A command represents intent to change state. One handler per command. Returns a result or void.

  • IndexContent — store and embed a piece of content
  • CreateTask — create a task from conversation or email
  • ClassifyEmail — run classification pipeline on an email
  • SpawnAgent — create a sub-agent for a subtask
  • ExecuteCode — run code in sandbox

Queries (read operations)

A query retrieves data without side effects. One handler per query.

  • SearchMemory — vector + FTS hybrid search
  • GetInboxSummary — aggregate email state
  • GetActiveContext — retrieve relevant context for agent turn
  • ListSkills — enumerate available SKILL.md files

Notifications (events, fan-out)

A notification is a domain event. Zero or many handlers subscribe. This is the primary decoupling mechanism between features.

  • MessageReceived → memory indexes it, proactive layer evaluates it
  • ContentIndexed → entity extraction runs, dashboard updates
  • TaskCompleted → predictive engine updates, notifications fire
  • EmailSynced → classification runs, task extraction runs

Pipeline Behaviors

Behaviors wrap every request, forming an onion around every command/query. They execute in order.

  Inbound Request
      │
      ▼
  ┌─────────────────────────────────────────┐
  │  1. Exception Handling                  │  catch all, return clean errors
  │  2. Logging + Correlation ID            │  structured trace, timing
  │  3. Authentication                      │  verify identity, inject user
  │  4. Validation                          │  schema + business rules
  │  5. Capability Gate                     │  check permissions for context
  │  6. Audit                               │  immutable audit log write
  │  7. Transaction                         │  wrap in DB transaction
  │  8. Retry / Circuit Breaker             │  transient failure handling
  │  9. Metrics                             │  timing, tokens, cost
  │                                         │
  │  ┌───────────────────────────────────┐  │
  │  │          Handler                  │  │
  │  │     (your business logic)         │──┼──▶ Notifications (fan-out)
  │  └───────────────────────────────────┘  │
  └─────────────────────────────────────────┘

Behavior Details

#BehaviorPurposeSource
1ExceptionBehaviorCatches unhandled errors, returns clean responsesEmailPipeline
2LoggingBehaviorCorrelation ID, request params, timing, error loggingEmailPipeline
3AuthBehaviorJWT / session token → user identity injectionEmailPipeline
4ValidationBehaviorSchema validation after auth mutationEmailPipeline + AgentClaw
5CapabilityGateBehaviorGranular permission checks: channel, persona, actionAgentClaw
6AuditBehaviorAppend-only audit log for all capability decisionsAgentClaw
7TransactionBehaviorDB transaction wrap: commit on success, rollback on errorEmailPipeline
8RetryBehaviorRetry with backoff for external API callsAgentClaw
9MetricsBehaviorToken usage, cost tracking, timingAgentClaw + OpenClaw

Plugin Extensibility

Skills and plugins can register custom behaviors at specific pipeline positions. A security plugin might add an extra classification behavior. A compliance plugin might add data-classification before the transaction layer.

This replaces hook chains (OpenClaw's current pattern) with something typed, ordered, and testable.
