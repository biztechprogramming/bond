11 — Configuration

Simple by default, powerful when you need it. The app does the right thing so you don't have to.

The Problem with OpenClaw's Config

Pain points we're fixing

  • Flat JSON wall — one huge openclaw.json with deeply nested keys. Hard to know what matters and what's optional.
  • Secrets in config files — API keys and tokens sit in plain text JSON alongside non-sensitive settings. Easy to accidentally commit to git.
  • Too many knobs on day one — a new user has to understand channels, agents, models, sandbox modes, DM policies, and more before anything works.
  • No guided setup — you read docs, then edit JSON by hand. If you get a key wrong, you find out at runtime.
  • Environment variables compete with config — some settings come from env, some from JSON, some from either. Unclear which wins.

Bond's Approach: Three Tiers of Configuration

Not all config is equal. API keys aren't the same kind of setting as sandbox memory limits. Bond separates configuration into three tiers based on who needs it and when.

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │   Tier 1: Setup Wizard                                      │
  │   "Get Bond running in 2 minutes"                           │
  │   API keys, model choice, first channel                     │
  │   ───────────────────────────────────────────                │
  │                                                             │
  │   Tier 2: Settings UI                                       │
  │   "Customize Bond to your liking"                           │
  │   Channels, agents, sandboxes, email, skills                │
  │   ───────────────────────────────────────────                │
  │                                                             │
  │   Tier 3: Config File                                       │

  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  Changes at any tier sync to the others.
Tier 1

Setup Wizard — "Just get me running"

First-run experience. A guided flow that asks only what's necessary to start. Everything else gets smart defaults.

1
Pick your AI provider Buttons: Anthropic / OpenAI / Google / Local (Ollama) / Other. One click.
2
Enter your API key Single text field. Bond validates it live (calls the API with a tiny request). Green check = you're good. Bond encrypts and stores it automatically — it never touches a config file.
3
Pick a model (optional) Dropdown pre-filtered to models available with your key. Default is pre-selected (Claude Sonnet for Anthropic, GPT-4o for OpenAI). Skip = use the default.
4
Connect a channel (optional) "Where do you want to talk to Bond?" Buttons: Web (already works), Telegram, Discord, etc. Each has a 3-step sub-flow. Skip = just use webchat.
5
Done Bond is running. Webchat opens. Everything else is defaults. The user configured 1-2 things.
💡 The wizard also works from CLI

bond setup runs the same flow as an interactive terminal wizard. Headless servers get the same easy experience.

Tier 3

Config File — "Give me full control"

~/.bond/bond.json for power users who want to version-control their config, copy it between machines, or edit in their editor. Every setting from the UI is represented here.

But secrets are never in this file.

Secret Management

This is the biggest change from OpenClaw. Secrets get their own system — they're never mixed with regular config.

How Bond handles secrets

  1. Secrets are stored in an encrypted vault — ~/.bond/data/credentials.enc, encrypted with Fernet. Not in bond.json. Not in environment variables (though env vars are accepted as an override for CI/Docker).
  2. The vault key lives in your OS keychain (macOS Keychain, Linux Secret Service, Windows Credential Manager). On first run, Bond generates a key and stores it there. No manual key management.
  3. Secrets are entered via UI or CLI — bond secret set ANTHROPIC_API_KEY sk-ant-... or the API Keys settings page. Both encrypt and store.
  4. Secrets are validated on entry — Bond makes a lightweight API call to verify the key works before saving. Immediate feedback.
  5. Secrets are referenced by name in config — bond.json says "model_provider": "anthropic", not the actual key. Bond resolves the key at runtime from the vault.
  6. Secrets are masked everywhere — Settings UI shows sk-ant-****...****1234. Logs never print secrets. The API never returns them.
  7. Secrets can be rotated — update via UI or CLI. Old value is overwritten. No manual file editing.
  User enters API key
      │
      ▼
  Validate (call provider API)
      │ ✓ valid
      ▼
  Encrypt (Fernet, key from OS keychain)
      │
      ▼
  Store in ~/.bond/data/credentials.enc
      │
      ▼
  bond.json just says: "provider": "anthropic"
      │ (no secrets in config file)
      ▼
  At runtime: Bond reads provider name → looks up key from vault → injects into LLM client

Environment variable override

For Docker / CI / headless setups, secrets can also come from environment variables. Env vars take priority over the vault when present. This is the only way secrets touch env — never as the primary storage mechanism for interactive users.

# Docker: pass secrets as env vars
docker run -d \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -e TELEGRAM_BOT_TOKEN=123456:ABC... \
  bondai/bond:latest

# Bond checks: env var → vault → not configured

The Config File (bond.json)

Clean, minimal, no secrets. A new user's config after the wizard might be just this:

{
  "model": {
    "provider": "anthropic",
    "default": "claude-sonnet-4-20250514"
  }
}

That's it. Everything else is defaults. As the user customizes through the UI, the file grows:

{
  "model": {
    "provider": "anthropic",
    "default": "claude-sonnet-4-20250514",
    "thinking": "low"
  },

  "channels": {
    "telegram": {
      "enabled": true
      // bot token is in the vault, not here
    },
    "discord": {
      "enabled": true,
      "sandbox": "minimal"
    }
  },

  "agents": {
    "coder": {
      "model": "anthropic/claude-sonnet-4-20250514",
      "sandbox": "coding"
    }
  },

  "sandboxes": {
    "coding": {
      "packages": ["nodejs", "python3", "git"],
      "network": "lan",
      "filesystem": "workspace"
    }
  }
}
💡 Only non-default values are written

Bond doesn't dump every possible key into the config. If you haven't changed it from the default, it's not in the file. This keeps configs small, readable, and diffable. bond config show --full shows all resolved values (defaults + overrides) for debugging.

Config Validation & Feedback

Bond validates config aggressively and gives clear, actionable feedback.

At startup

  • Schema validation — every key is checked against a typed schema. Unknown keys produce a warning (not an error — forward compatibility). Wrong types produce an error with the expected type.
  • Dependency checks — if you enable Telegram but have no bot token in the vault, Bond tells you: "Telegram is enabled but no bot token found. Run: bond secret set TELEGRAM_BOT_TOKEN"
  • Credential validation — on first startup (or after a key change), Bond tests each credential. Failed keys get a clear error: "Anthropic API key is invalid (HTTP 401). Update it: bond secret set ANTHROPIC_API_KEY"

In the Settings UI

  • Live validation — fields validate as you type. API keys show a spinner → green check / red X.
  • Test buttons — "Test Connection" for every external integration (email, channels, LLM providers).
  • Dependency hints — if you enable a feature that requires something else, the UI tells you what's missing before you save.
  • Diff view — before applying changes, show what will change (like a git diff for your config).

CLI doctor

Config Sync Between Tiers

All three tiers (wizard, UI, file) read and write the same underlying state. There's no "the UI wrote one thing but the file says another" confusion.

  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │  Wizard  │     │ Settings │     │bond.json │
  │  (CLI/   │     │   UI     │     │ (file)   │
  │   Web)   │     │          │     │          │
  └────┬─────┘     └────┬─────┘     └────┬─────┘
       │                │                │
       └────────┬───────┴────────┬───────┘
                ▼                ▼
         ┌─────────────┐  ┌──────────────┐
         │ Config Store │  │ Secret Vault │
         │ (bond.json)  │  │ (encrypted)  │
         └──────┬──────┘  └──────┬───────┘
                │                │
                ▼                ▼
         ┌──────────────────────────────┐
         │    Runtime Config Object     │
         │  (merged: defaults + file    │
         │   + env overrides + vault)   │
         └──────────────────────────────┘

Merge order (last wins):

  1. Built-in defaults — hardcoded sensible values
  2. bond.json — user overrides
  3. Environment variables — for Docker/CI (format: BOND_MODEL_DEFAULT=claude-sonnet-4-20250514)
  4. Secret vault — credentials resolved by provider name

📝 Remember: secrets are never in this file

Notice there are no API keys, bot tokens, or passwords anywhere in bond.json. The file is safe to commit to git, share, or back up without risk. Secrets live in the encrypted vault.

CLI Secret Commands

# Set a secret (validates before storing)
bond secret set ANTHROPIC_API_KEY
# → prompts for value (hidden input), validates, encrypts, stores

# Or pass inline (for scripts)
bond secret set ANTHROPIC_API_KEY sk-ant-...

# List configured secrets (masked)
bond secret list
# ANTHROPIC_API_KEY    sk-ant-****...****a1b2   ✓ valid
# TELEGRAM_BOT_TOKEN   1234****...****ABCD      ✓ valid
# DISCORD_BOT_TOKEN    (not set)

# Remove a secret
bond secret remove TELEGRAM_BOT_TOKEN

# Validate all secrets
bond secret test
# ✓ ANTHROPIC_API_KEY — valid (Claude Sonnet 4 available)
# ✗ OPENAI_API_KEY — expired (HTTP 401)

# Export secrets for Docker (to stdout, ephemeral)
bond secret export --env
# ANTHROPIC_API_KEY=sk-ant-...
# TELEGRAM_BOT_TOKEN=123456:ABC...

Channel Setup Flows

Each channel has a streamlined setup flow in the Settings UI — not a wall of config keys.

Example: Telegram setup

1
Create a bot Link to @BotFather with step-by-step instructions shown inline. "Done? Paste your token below."
2
Paste bot token Bond validates it (calls getMe), shows the bot's name and username as confirmation. Encrypts and stores in vault.
3
Done Telegram is live. Default DM policy is "pairing" — safe. Optional: configure group settings, DM allowlist.

Every channel follows this pattern: guided instructions → paste credential → validate → done. Advanced settings are available but tucked away.

Migration from OpenClaw

For users coming from OpenClaw:

# Import OpenClaw config (extracts settings + migrates secrets to vault)
bond import openclaw ~/.openclaw/openclaw.json

# What it does:
# 1. Reads openclaw.json
# 2. Extracts secrets (API keys, bot tokens) → encrypts into Bond vault
# 3. Translates remaining config → bond.json format
# 4. Reports what was migrated and what needs manual attention
