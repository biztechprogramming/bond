Module Boundary Rules

Rule 1: Features never import features

The email module never imports from the tasks module. Communication is through mediator notifications only. This keeps features decoupled and independently testable.

Rule 2: Features depend on foundations, not the reverse

Features import from foundations/ (knowledge store, entity graph, etc.). Foundations never import from features. Foundations may import from other foundations.

Rule 3: The gateway handles transport + LLM, not business logic

The gateway handles transport (WebSocket, channels, sessions) and all LLM communication (auth, streaming, provider APIs via pi-agent-core/pi-ai). This preserves OAuth subscription support — the #1 priority feature. All other business logic (tools, mediator, knowledge, orchestration) lives in the Python backend. The gateway is a routing + LLM proxy layer.

Rule 4: API routes are thin

API routes in api/v1/ construct a mediator command/query and dispatch it. They don't contain logic — that's in handlers.

Rule 5: One handler per command/query

The mediator enforces this: each command type has exactly one handler. Multiple notification handlers are fine (fan-out).
