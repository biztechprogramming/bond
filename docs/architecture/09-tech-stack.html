Gateway ↔ Backend Communication

The TypeScript gateway and Python backend run as two processes, communicating via a local protocol:

  • Gateway → Backend: HTTP calls to FastAPI endpoints (agent turns, tool execution, queries)
  • Backend → Gateway: HTTP calls for LLM completion requests; WebSocket connection for receiving streamed LLM responses, pushing events (notifications, status updates)
  • Shared: SQLite database (single file, WAL mode for concurrent reads)

In Docker, both processes run in the same container managed by a simple process supervisor. In development, they run separately with hot reload.

Why Docker socket mount for sandboxes?

Bond manages sandbox containers from inside its own container by mounting the host's Docker socket. This is simpler and more performant than Docker-in-Docker (DinD). The tradeoff: the Bond container has Docker access on the host — but it already needs this for sandbox management. For users who don't want socket mounting, we'll support a DinD sidecar as an alternative.

Development Setup

# Clone
git clone https://github.com/bond-ai/bond.git
cd bond

# Backend
uv sync                        # install Python deps
uv run migrate up              # run database migrations
uv run dev                     # start FastAPI with hot reload

# Gateway (separate terminal)
cd gateway
pnpm install
pnpm dev                       # start gateway with hot reload

# Frontend (separate terminal)
cd frontend
pnpm install
pnpm dev                       # start Next.js dev server

# Or, all at once:
make dev                       # runs all three with process manager

# First run? Setup wizard handles API keys + config:
bond setup                     # interactive setup (secrets → encrypted vault)

Production Setup

# Docker (recommended for end users)
docker run -d \
  -p 18789:18789 \
  -v ~/.bond:/home/bond/.bond \
  -v /var/run/docker.sock:/var/run/docker.sock \
  bondai/bond:latest

# Or native (for developers)
bond start                     # starts gateway + backend + frontend
