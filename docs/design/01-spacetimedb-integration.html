13 — SpacetimeDB Integration

Evolving from passive SQLite storage to an active, real-time relational engine.

Overview

While the initial design used SQLite for simplicity, SpacetimeDB offers a "Serverless Real-Time" paradigm where application logic (Reducers) lives directly inside the database. This eliminates the traditional mismatch between database state and application memory, providing instant reactivity for multi-agent coordination and UI updates.

Why SpacetimeDB?

  • Unified State: No more "database vs. runtime memory." The database is the runtime.
  • Instant Reactivity: Clients (Webchat, CLI) subscribe to tables. When an agent updates a row, the UI reflects it immediately without polling.
  • ACID Reducers: Logic like "deducting credits" or "claiming a task" is transactional and runs in WebAssembly inside the DB.
  • Zero-Downtime Migrations: Module hotswapping allows updating Bond's "brain" logic without dropping active agent sessions.

Architectural Shift

The Python backend (FastAPI) shifts from being the owner of the state to a client of the SpacetimeDB module. The "Mediator Pipeline" logic can gradually migrate into Reducers.

  [ Channels / UI ] ◀──(Subscription)──▶ [ SpacetimeDB Module ]
                                               │
                                       (Logic & Tables)
                                               │
  [ Python Worker ] ◀──(Reducer Calls)──▶ [ WASM Reducers ]
          │                                    │
          └─▶ (Heavy lifting: LLM, Embeddings, Tool Execution)

Database Modules

Bond's SpacetimeDB module will be written in Rust (primary) or TypeScript, compiled to WASM.

Tables (Subscribable)

  • Agents — Status, current task, active tool, heartbeat.
  • Messages — Full conversation history with real-time tailing.
  • TaskGraph — Real-time dependency tracking and status updates.
  • ActiveSubscriptions — Managed OAuth/API profiles and their cooldowns.

Reducers (Transactional)

  • SendMessage(content, sender_id) — Validates sender, appends message, triggers agent notification.
  • UpdateTask(task_id, new_status) — Atomically updates status and checks if dependencies are cleared.
  • LogAudit(action, actor, context) — Immutable append-only audit trail.
  • ClaimTask(agent_id, task_id) — Ensures atomic task assignment in multi-agent race conditions.

Migration Strategy

  1. Hybrid Storage: Keep SQLite for local file-heavy data (Blobs, Vector Indexes) while moving coordination state (Tasks, Agents, Messages) to SpacetimeDB.
  2. Client SDKs: Use the SpacetimeDB SDK in the TypeScript Gateway for instant UI updates.
  3. Incremental Logic: Move validation logic from Python/FastAPI into WASM Reducers for guaranteed consistency.

Vector Search Note

As SpacetimeDB does not currently have native vector indexing as robust as sqlite-vec, we will maintain a **Sidecar Vector Store** (SQLite) for RAG/Embeddings, linked to SpacetimeDB records via unique IDs.
