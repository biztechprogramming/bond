# Design Doc 079 — Deployment Remaining Gaps

**Status:** Draft
**Depends on:** 077, 078
**Date:** 2026-03-27

## What Was Implemented

### Gateway (backend)
- **Deployment Runs** (`runs.ts`): In-memory run tracking with create/list/get/cancel. Each run goes through queued → running → success/failed/cancelled.
- **SSE Streaming**: `GET /runs/:id/stream` and `POST /execute-plan` both stream step progress, log lines, and completion events via Server-Sent Events with 15s heartbeat.
- **Rollback Endpoint**: `POST /rollback` creates a rollback-type run.
- **Run Lifecycle**: Runs execute 6 simulated steps (validate, build, push, deploy, health, monitor) and write a receipt on completion.

### Frontend (deploy UI)
- **DeployPage**: Added "Deployments" tab showing all recent runs via `DeploymentRunsList`.
- **OneClickShipWizard**: Added "Allocate" step (step 3 of 4) that fetches suggested port/directory defaults, shows editable fields, performs real-time conflict checking, and saves the allocation before deployment.
- **ShipProgress**: Wired to real SSE from `/execute-plan`, with proper event parsing, inline log display, cancel button, and graceful fallback to simulated progress.
- **AppDashboard**: Added "Recent Deployments" section with rollback buttons.
- **AppDetail**: Added deployment runs in overview tab with rollback support.
- **DeploymentRunsList**: New reusable component for displaying runs with status, logs, and rollback actions.

### Design Doc 077 Integration
- Allocation step in wizard calls `POST /allocations/suggest` and `POST /allocations/check-conflicts`.
- Allocation is saved via `POST /allocations` before the deployment ships.
- Conflict detection shows port collisions and directory overlaps inline.

## What Still Doesn't Work

### Critical — Requires Real Infrastructure

| Gap | Description | Effort |
|-----|-------------|--------|
| **Remote SSH execution** | `executeDeploymentRun` simulates steps with timers. Real deployments need SSH2 connection to target servers, SCP upload of scripts, and stdout/stderr streaming. | 3–5 days |
| **Real build pipeline** | No actual `docker build`, `git clone`, or artifact push occurs. The pipeline-executor runs locally on the gateway host. | 3–5 days |
| **Health checks** | Health check step is simulated. Needs HTTP/TCP probes against actual service endpoints using allocation data. | 1–2 days |
| **Rollback execution** | Rollback creates a run but doesn't execute the rollback script generated by quick-deploy. Needs SSH execution of the rollback script. | 1–2 days |

### High Priority — UI/API Gaps

| Gap | Description | Effort |
|-----|-------------|--------|
| **Live Logs from servers** | `LiveLogViewer` exists but doesn't connect to real log sources. Needs log collection agent or SSH tail on target. | 2–3 days |
| **Component status tracking** | `component_status` table from doc 078 not implemented. Health/deploy status per component not persisted. | 1 day |
| **Promotion gate UI** | Promotion API exists but no frontend gate panel (approve/reject buttons). | 1–2 days |
| **Receipt browser** | No receipt list UI with filtering/pagination. Data exists in filesystem. | 1 day |
| **Add Component form** | Quick action to add a component has no form. | 0.5 day |
| **Agent Settings panel** | Quick action navigates nowhere. Need UI for agent pause/resume/abort. | 0.5 day |

### Medium Priority — Polish

| Gap | Description | Effort |
|-----|-------------|--------|
| **Run persistence** | Runs are in-memory only. Gateway restart loses all run history. Need file or DB persistence. | 0.5 day |
| **Cron trigger scheduling** | Triggers are stored but never actually scheduled (no node-cron or similar). | 1 day |
| **Auto-rollback** | No automatic rollback on health failure. Needs health check integration + rollback trigger. | 1–2 days |
| **Environment comparison** | Compare endpoint exists but no frontend integration. | 0.5 day |
| **Pipeline-as-code UI** | YAML parser exists, no management UI beyond the editor. | 1 day |
| **Webhook receiver** | Push webhook exists but doesn't trigger deployments automatically. | 0.5 day |
| **Server metrics** | CPU/RAM/Disk in InfraMap rely on probe data that's never populated. | 1–2 days |

## Prioritized Implementation Order

1. **SSH remote execution** — This is the single biggest gap. Without it, deployments are simulated. Implement using `ssh2` npm package with connection pooling.
2. **Run persistence** — Write runs to `~/.bond/deployments/runs/` as JSON. Load on startup.
3. **Real health checks** — HTTP GET to `health_endpoint` from allocation's service port assignments.
4. **Rollback execution** — Find rollback script from script registry, execute via SSH.
5. **Live log streaming** — SSH tail -f on log files from allocation's `log_dir`.
6. **Component status** — Persist health/deploy status per component after each run.
7. **Promotion gate UI** — Frontend panel wired to existing promotion API.
8. **Everything else** — Receipt browser, agent settings, cron scheduling, etc.

## Architecture Notes

The current simulated execution (`runs.ts`) is designed to be replaced. The `executeDeploymentRun` function should be swapped for a real implementation that:

1. Resolves the target server from `resource_id` (SpacetimeDB resource table)
2. Establishes SSH connection using stored credentials
3. Uploads the deploy script (from script registry) via SFTP
4. Executes the script, streaming stdout/stderr through the existing EventEmitter
5. Runs health check probes against the allocation's health endpoints
6. Writes receipt with real timing and output data

The SSE streaming infrastructure, run tracking, and frontend are all ready for this — only the execution backend needs to change.
