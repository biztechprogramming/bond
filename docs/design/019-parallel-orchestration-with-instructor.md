# Design Doc 019: Parallel Tool Orchestration with Instructor

## Overview
This document outlines the implementation of a "High-Power Orchestration, Low-Power Execution" pattern for Bond. 
An expensive model (e.g., Claude 3.5 Sonnet / GPT-4o) will be used to analyze complex tasks and decompose them into a structured parallel execution plan. A fleet of cheaper models (e.g., Haiku / GPT-4o-mini) will then execute the resulting tool calls in parallel.

## Goals
1. **Performance**: Significantly reduce wall-clock time for complex tasks (e.g., multi-file refactors, broad searches).
2. **Cost Efficiency**: Use high-powered reasoning only for the "Architect" phase (1-2 calls), delegating the "Worker" phase to inexpensive models.
3. **Reliability**: Use **Instructor** to ensure the Architect produces a strictly valid execution graph that can be parsed and dispatched without retries.

## Architecture

### 1. The Orchestration Schema
We will define a `ParallelWorkPlan` using Pydantic. This model forces the Architect to think through the dependencies and identify which tool calls can run safely in parallel.

```python
from pydantic import BaseModel, Field
from typing import List, Dict, Any, Optional

class ToolInvocation(BaseModel):
    tool_name: str = Field(..., description="The name of the tool to call")
    arguments: Dict[str, Any] = Field(..., description="The arguments for the tool")
    model_override: Optional[str] = Field(None, description="Optional cheaper model to use for this specific call")
    description: str = Field(..., description="Briefly what this specific call is doing")

class ParallelBatch(BaseModel):
    batch_name: str
    calls: List[ToolInvocation] = Field(..., min_items=1)
    
class ParallelWorkPlan(BaseModel):
    reasoning: str = Field(..., description="High-level reasoning for the chosen decomposition")
    batches: List[ParallelBatch] = Field(..., description="Sequential batches of parallel calls. Batch 2 starts only after Batch 1 finishes.")
```

### 2. The Execution Flow
1. **Architect Phase**: 
   - Input: User request + Context.
   - Model: `BOND_ARCHITECT_MODEL` (High-power).
   - Tool: `instructor.patch()` with `response_model=ParallelWorkPlan`.
2. **Dispatch Phase**:
   - For each `ParallelBatch` in `ParallelWorkPlan.batches`:
     - Launch all `ToolInvocation` calls in the batch simultaneously using `asyncio.gather()`.
     - Route each call to the `BOND_WORKER_MODEL` (Low-power) unless overridden.
3. **Synthesis Phase**:
   - Collect all results from the batch.
   - If more batches remain, provide the results as context for the next batch.
   - Final response generated by the Architect or Worker model.

### 3. Native Parallel Optimizations
Beyond the general orchestration loop, specific high-latency read operations will be natively parallelized:
- **Parallel File Discovery & Reading**: When the Architect requests multiple file reads or an entire directory scan, the `NativeRegistry` will utilize `asyncio.gather` to perform I/O concurrently rather than sequentially.
- **Concurrent SpacetimeDB Schema Reflection**: Parallelizing the retrieval of table structures and reducer signatures from the SpacetimeDB module via the HTTP/WS control plane, ensuring the agent is immediately oriented with the latest global state.
- **Parallel Context Retrieval**: Searching across multiple memory types (Knowledge, Entity Graph, SpacetimeDB Message History) will be dispatched concurrently during the `ContextDistillation` phase.

## Implementation Plan

### Phase 1: Worker Core Update (`backend/app/agent/loop.py`)
- Introduce an `execute_parallel_batch` helper.
- Update the main loop to recognize a "Parallel Request" intent.

### Phase 2: Tool Registry Enhancement
- Ensure the `NativeRegistry` is thread-safe/async-safe for concurrent execution.
- Add `model_override` support to the tool execution context.

### Phase 3: UI Feedback
- Update the Gateway and Frontend to handle multiple concurrent `tool_call` SSE events.
- Ensure the UI can display "Parallel Execution" progress bars or grouped logs.

## Security & Safety
- **Concurrency Limits**: Implement a `MAX_CONCURRENT_TOOLS` (default: 5) to prevent rate-limit exhaustion or local resource starvation.
- **Side Effects**: The Architect must be prompted to avoid parallelizing tools with conflicting side-effects (e.g., two parallel `file_write` calls to the same file).

## Success Criteria
- [ ] Complex multi-file search tasks complete 3x faster.
- [ ] Architect successfully delegates at least 80% of tool-call volume to the cheaper worker model.
- [ ] Instructor validation catches and corrects 100% of malformed parallel plans before execution.
