## Data Loss Incident
- `--delete-data=on-conflict` on `spacetime publish` wiped ALL SpacetimeDB data (not just conflicting tables)
- Lost agents, providers, models, API keys, conversations, settings
- **NEVER use --delete-data in migrate script** — removed it, requires manual CLI for destructive migrations
- Re-seeded from SQLite (knowledge.db still has agents, providers, models, API keys, aliases)
- Created scripts/seed-spacetimedb.sh for future re-seeding
- SpacetimeDB SQL INSERT doesn't support partial columns for tables with .optional() fields — use `spacetime call` for those
- Provider table: apiBaseUrl and modelsEndpoint are .optional() — can't INSERT via SQL, must use reducer

## Agent Channel Save Bug
- INSERT INTO agent_channels was missing `id` and `created_at` columns — SpacetimeDB silently rejected
- Column ORDER matters in SpacetimeDB SQL INSERT — must match table definition order (id, agentId, channel, sandboxOverride, enabled, createdAt)
- stdb.query() was swallowing write errors (returned [] instead of raising) — fixed to raise RuntimeError on INSERT/UPDATE/DELETE failures
- Added global exception handler in backend/app/main.py — ALL unhandled exceptions return structured JSON

## SpacetimeDB Consistency & Subscriptions
- SpacetimeDB has eventual consistency on reads after writes via HTTP SQL
- **User strongly wants subscriptions for all data** — not REST polling
- Wired up AgentsTab to use SpacetimeDB subscriptions (useSpacetimeDB hook)
- Added subscriptions for: agents, agent_channels, agent_workspace_mounts, llm_models, providers, provider_aliases
- SDK uses snake_case property names (agent_channels not agentChannels, llm_models not llmModels)
- Row data from subscriptions uses camelCase field names (displayName, agentId, etc.)
