🛠️ Proposal: Agent Temperature Configuration

Adding per-agent temperature control to fix GPT-5 compatibility and give users fine-grained control

Problem: Temperature is hardcoded to 0.7 in the agent loop. GPT-5 models only support temperature=1, causing litellm.UnsupportedParamsError on every conversation start.

Solution: Add a temperature field to the agent configuration NEW FIELD with smart defaults — when set to "Auto", the backend picks the right value for the model (1.0 for GPT-5, 0.7 for others). Users can also override manually.

GPT-5 models only support temperature=1. Enable "Auto" to let Bond pick the right value automatically, or set to 1.0 manually. For GPT-5.1, temperature is supported only when reasoning_effort='none'.
Controls randomness in model responses. Lower = more deterministic, higher = more creative. "Auto" uses 0.7 for most models, 1.0 for GPT-5, and omits the parameter for reasoning models.

How It Works — Scenarios

Scenario 1: GPT-5 with "Auto" enabled ✅

User selects openai/gpt-5 as the model and checks "Auto".
→ Backend detects GPT-5 model, sends temperature=1.0 (the only supported value).
→ No error. Conversation starts normally.

Scenario 2: Claude with custom temperature

User selects anthropic/claude-sonnet-4-20250514 and sets temperature to 0.3.
→ Backend sends temperature=0.3 to litellm.
→ More deterministic, focused responses for coding tasks.

Scenario 3: GPT-5 with manual override to 0.7 ⚠️

User selects openai/gpt-5 and manually sets 0.7.
→ The warning banner appears: "GPT-5 only supports temperature=1".
→ Backend applies litellm.drop_params as a safety net, so the call still succeeds (litellm strips the unsupported param), but the UI warns the user it will be ignored.

Scenario 4: Existing agents (migration)

Existing agents have no temperature value → treated as null → "Auto" mode.
→ Behavior: 0.7 for Claude/GPT-4o (same as today), 1.0 for GPT-5.
→ Zero breaking changes for existing configurations.
